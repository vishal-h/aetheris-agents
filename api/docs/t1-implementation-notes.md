# T1 Implementation Notes — TAP Plumbing

## at1cmd.exs is the OrbConfig entry point

The milestone sprint case passes three files to `run_orb`:
```
at1cmd.exs  cot1_stub.exs  at1qry.exs
```

The Aetheris CLI (`run_attached` in `run.ex`) takes `[path | _rest]` — it loads only the first file and ignores the rest. The three-file invocation is documentary, not functional.

**Decision**: `at1cmd.exs` returns an `OrbConfig` containing all three `RunConfig` definitions. `cot1_stub.exs` and `at1qry.exs` are standalone `RunConfig` files used only for the individual `Code.eval_file` sanity checks in the agentic instructions. They are not involved in the sprint execution.

T2 should keep this structure. The sprint case for T2 should continue to pass `at1cmd.exs` first.

## sandbox_path goes up two levels to api/

All three agent files are nested two directories deep from `api/` (e.g., `api/tenant/agents/`). The sandbox_path formula:

```elixir
agent_root = Path.expand(Path.join(Path.dirname(__ENV__.file), "../.."))
```

This gives all three agents the same sandbox root (`api/`), enabling:
- at1cmd: `tenant/scripts/parse_csv.py`, `tenant/data/sample_enrollments.csv`
- cot1_stub: `gateway/scripts/validate_intent.py`, `domain/ct.stu.vocabulary.jsonl`
- at1qry: `tenant/scripts/gap_analysis.py`

The payslip orchestrator goes up one level because its scripts are in `payslip/scripts/`. Here we go up two because the tenant and gateway subdirectories split the scripts from the domain docs.

## Intent ID flows through send_message

The intent_id is generated by `package_intent.py` at runtime and is not knowable when the OrbConfig is constructed. It flows through the orb via messages:

1. at1cmd runs package_intent.py → extracts `intent_id` from output JSON
2. at1cmd writes blackboard key `tap:intent:{intent_id}`
3. at1cmd sends message to cot1_stub with `intent_id` in the message body
4. cot1_stub waits for `message_received`, extracts intent_id, reads blackboard
5. cot1_stub runs scripts, writes `tap:result:{intent_id}`
6. cot1_stub sends message to at1qry with intent_id
7. at1qry waits for `message_received`, reads `tap:result:{intent_id}`, runs gap_analysis

Run IDs are pre-established from `orb_id` in the OrbConfig so `send_message` targets are known at prompt-write time.

## validate_intent: termName is required but defaultable

The vocabulary doc lists `termName` in the `requires` array for `enroll_students`. It also has a rule:

```jsonl
{"record_type": "rule", "intent": "enroll_students", "field": "termName", "if_missing": "use_default", "default_from": "current_term", ...}
```

The validator checks rules for `if_missing: "use_default"` before applying required-field checks. Fields with this rule are excluded from the required check — their absence is not an error. This is the correct behaviour: cot1 will fill termName from the `current: true` term lookup (Annual) when it's absent from the payload.

## Ravi Kumar is the non_idempotent test case

Row 3 in `sample_enrollments.csv` has no DOB and no admissionNumber. `stub_cot1.py` detects this and assigns `identity_state: "non_idempotent"` (random UUID v4). `gap_analysis.py` catches this and emits a gap with `suggested_action: "add DOB or admissionNumber to source data"`.

The acceptance criterion — "at1qry produces a gap report identifying the Ravi Kumar row as non_idempotent" — is covered by this chain.

## Sprint case path deviation from milestone spec

The milestone sprint case uses paths like `uc-api-agent/tenant/...`. The actual folder in `aetheris-agents/` is `api/`, not `uc-api-agent/`. Sprint runs from the `aetheris/` repo so the correct paths are `../aetheris-agents/api/...`.

The sprint case added to `sprint.sh` uses the corrected paths. The milestone spec should be treated as aspirational on this detail.

## context_strategy: :full for short-lived orb agents

The initial implementation used `context_strategy: :rolling, max_context_steps: 6` on cot1_stub and at1qry, following the guide's recommendation for orchestrators. This caused an HTTP 400 on the first sprint run:

> unexpected tool_use_id found in tool_result blocks

cot1_stub ran 7 steps. At step 6, the rolling window dropped the oldest messages, splitting a tool_use/tool_result pair. The API rejected the resulting context because it contained a tool_result referencing a tool_use that had been truncated away.

**Fix:** `context_strategy: :full` (no `max_context_steps`) for both cot1_stub and at1qry.

The guide's rolling recommendation is for long-running orchestrators that spawn many sub-agents over 15–20 steps. Short-lived pipeline agents (under 10 steps, no large file reads) have no overflow risk and must use `:full` to keep tool_use/tool_result pairs intact.

**Rule for T2 and beyond:** use `:rolling` only when `max_steps` significantly exceeds 10 and the agent accumulates large context per step. For agents that do a handful of tool calls and report, use `:full`.

## What T2 should know

**Domain docs are enroll_students-only stubs.**
- `domain/ct.stu.vocabulary.jsonl` has `intent_count: 1` and no `setup_institution` or `setup_courses` sections.
- `domain/ct.stu.behaviour.jsonl` has only the ETL-mode execution steps for enroll_students.
- T2 extends both docs with the setup intents and their direct-mode execution/outcome records.

**cot1_stub.exs is replaced by cot1.exs in T2.**
`cot1_stub.exs` returns a mock result with `status: "queued"` and a fake `job_ref`. T2 writes `gateway/agents/cot1.exs` which calls ct-api via `gateway/scripts/build_etl_job.py` and `gateway/scripts/submit_etl_job.py`. The OrbConfig in `at1cmd.exs` should be updated in T2 to reference the real `cot1.exs` RunConfig inline.

**The sprint case only uses at1cmd.exs.**
T2's sprint case can pass `at1cmd.exs` as a single argument to `run_orb` (or keep the three-file form — both work due to the CLI behaviour). The T2 `at1cmd.exs` OrbConfig should swap out the cot1_stub RunConfig for the cot1 RunConfig.

**Python tests use python3 -m pytest.**
The mise-managed Python 3.12 requires `python3 -m pip install pytest` — the system `pip3` (Python 3.10) installs to a different site-packages. The sprint case calls `python3 -m pytest` which is correct.
