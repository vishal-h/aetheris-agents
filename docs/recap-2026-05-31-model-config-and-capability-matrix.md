## 1. Model/provider configuration (both repos)

### Goal

Replace hardcoded `"anthropic"` and `"claude-haiku-4-5-20251001"` strings in
every agent `.exs` file with a two-level env-var fallback chain so models can
be swapped without editing code.

### What was built

**`aetheris-agents/.env`** (committed) — project-wide defaults:

```bash
AETHERIS_MODEL=claude-haiku-4-5-20251001
AETHERIS_PROVIDER=anthropic

# Per-use-case overrides (uncomment to activate):
# PROVENANCE_MODEL=claude-haiku-4-5-20251001
# PAYSLIP_MODEL=...
# EMAIL_MODEL=...
# DRIVE_MODEL=...
# API_MODEL=...
```

**`aetheris-agents/.env.example`** — template for `.env.local` local overrides
(`.env.local` is gitignored).

**`aetheris/scripts/sprint.sh`** — sources `.env` before running any agents:

```bash
AGENTS_DIR="$(cd "$(dirname "$0")/../aetheris-agents" 2>/dev/null && pwd)"
if [ -f "$AGENTS_DIR/.env" ]; then
  set -a && source "$AGENTS_DIR/.env" && set +a
fi
```

**All 22 agent `.exs` files** — updated to use the fallback pattern:

```elixir
model    = System.get_env("PROVENANCE_MODEL") || System.get_env("AETHERIS_MODEL") || "claude-haiku-4-5-20251001"
provider = System.get_env("AETHERIS_PROVIDER") || "anthropic"
```

Prefix per use case: `PROVENANCE_MODEL`, `PAYSLIP_MODEL`, `EMAIL_MODEL`,
`DRIVE_MODEL`, `API_MODEL`. Root `agents/capability_matrix_*.exs` use
`AETHERIS_MODEL` directly (no use-case prefix).

## 2. Bug: Application.get_env returns nil at agent eval time

### What happened

First attempt used `Application.get_env(:aetheris, :default_model)` as the
mid-tier fallback (via `config/runtime.exs`). The provenance capability matrix
sub-agent failed immediately with:

```
error: "unknown provider: \"\""
model => nil
```

### Root cause

`mix aetheris run` calls `load_agent_file` (which evaluates the `.exs` script
via `Code.eval_file`) **before** `ensure_started` (which calls
`Application.ensure_all_started`). The Application env is not yet populated
when agent scripts run, so `Application.get_env` always returns `nil`.

Confirmed by reading `aetheris/lib/aetheris/cli/commands/run.ex`:

```elixir
with {:ok, config} <- RunHelpers.load_agent_file(path),   # .exs evaluated here
     {:ok, config} <- apply_overrides(config, opts),
     :ok            <- RunHelpers.ensure_started(),          # app starts here
     ...
```

### Fix

Replace `Application.get_env` with `System.get_env` throughout all agent files.
System env vars are always available. The dead `config/runtime.exs` lines were
also removed:

```elixir
# Removed from aetheris/config/runtime.exs — never reached:
# config :aetheris, :default_model,    System.get_env("AETHERIS_MODEL")    || "..."
# config :aetheris, :default_provider, System.get_env("AETHERIS_PROVIDER") || "..."
```

### Secondary bug: duplicate env var lookup

The sed substitution that applied the fix produced a duplicate for the 7
`agents/capability_matrix_*.exs` files (which already had `AETHERIS_MODEL` as
their use-case prefix):

```elixir
# Bug:
model = System.get_env("AETHERIS_MODEL") || System.get_env("AETHERIS_MODEL") || "..."
# Fixed:
model = System.get_env("AETHERIS_MODEL") || "..."
```

## 3. Capability matrix: .exs files silently skipped from scripts/

### Problem

The `capability_matrix_provenance.exs` system prompt instructed the sub-agent
to collect only `.py` filenames from `{use_case}/scripts/`. This silently
skipped `provenance/scripts/register_eval_task.exs` (and any future `.exs`

### Fix (applied to all 6 sub-agents consistently)

Step 3 collect line changed from:

```
Collect .py filenames only. Skip __init__.py and conftest.py.
```

to:

```

```

Step 4 header updated from `For each .py script:` to `For each .py or .exs script:`,
with a note that `.exs` purpose comes from the opening `#` comment rather than
a docstring.

`docs/capability-matrix-runbook.md` was also updated:

- "When to re-run" trigger now mentions `.exs` scripts
- Directory structure note updated to `.py and .exs`
- Step 2 note warns against reverting the `.exs` collection when copying the
  template for a new use case

## Key design decisions

**Why System.get_env, not Application.get_env**
Agent `.exs` files are evaluated by `Code.eval_file` before the OTP application
starts. `System.get_env` is always available; `Application.get_env` returns `nil`
at eval time.

**Why .env is committed**
The `.env` file contains no secrets — only model names and provider strings. It
serves as the project-wide default that all developers and CI share. Machine-
specific overrides go in `.env.local` (gitignored).

**Why sprint.sh sources .env**
`mix aetheris run` run standalone inherits the caller's shell env. `sprint.sh`
runs in a subprocess that may not have `.env` sourced, so it sources it
explicitly before calling any agents.

**Hardcoded default as safety net**
The final `|| "claude-haiku-4-5-20251001"` in every agent ensures the agent
never receives a nil model even if both env vars are unset. It is a safety net,
not the primary configuration mechanism.
