## Findings

18. **[blocking]** The bound kills legitimately paused runs — and contradicts the design's own stated rationale. The notes and the config comment justify the 300 000 ms default as "the same evidence the sweep and the badge call dead." But the sweep explicitly does **not** call paused runs dead: `skipped_paused` is a named census category, and BL-003's paused-run exclusion was exercised with an unexpired `wait_for_event` fixture that *survives* the sweep. `handle_run_status`'s `_other` branch — where the bound now lives — receives `"paused"` like any non-terminal status, so a run legitimately parked on `wait_for_event` past the threshold, emitting nothing because waiting is its job, trips the bound. And the consequence is worse than a wrong error: the CLI and the run share the BEAM, so `exit 1` kills a run the harness considers alive — the pre-fix hang at least left the run's own `max_wait_ms` expiry to resolve it. The bound converts one failure mode (hang on a dead run) into a new one (kill a live one) for exactly the status the sweep exempts. Suggested fix, mirroring the sweep: exempt `"paused"` from the timeout branch — while paused, sleep-and-continue without consulting the clock (resetting `since` on the paused↔running transitions falls out of the status-in-the-key design for free); the run's own wait-expiry semantics then produce the activity or the terminal status. A stuck-forever paused run thereby hangs the CLI again — accepted, deliberately: that is the sweep's position too, and `wait_for_event`'s expiry is the mechanism that owns bounding a pause. State the exemption in the moduledoc and test it with an unexpired-pause fixture (paused past the threshold → no error; then resumed → completes).

19. **[non-blocking — chases finding 18]** The same-evidence-as-sweep claim appears in three places the fix must correct in the same round: the `config.exs` comment, the runbook's Configuration paragraph, and the impl notes' default-kept rationale. Post-18 the claim becomes true — "the CLI gives up on the same evidence the sweep calls dead, and exempts what the sweep exempts" — and should say so, including one runbook sentence on paused behavior (a paused run waits indefinitely under the CLI, bounded by its own `wait_for_event` expiry, not by this threshold). The "not a run timeout" subsection is good operator writing; this is its companion surprise.

20. **[non-blocking]** The "status change alone counts as activity" test uses `paused` as its intermediate status — after finding 18, `paused` becomes exempt-special rather than a neutral status value, and the test would pass for the wrong reason. Swap the intermediate transitions to non-paused statuses so the test keeps proving what its name claims, and let the new paused-exemption test own the paused semantics.

21. **[non-blocking, ratification]** The config-env mutation tests (`put_env`/`on_exit` restore) are `async: false`, which serializes the file — but the env key is process-global, and any *other* async test file calling `await_run` mid-run would read the mutated value. Full suite is green today, so no current collision; approved as-is, with a one-line comment in the two tests noting the global-env hazard so a future async caller of `await_run` in tests knows why these two are load-bearing `async: false`.

22. **[question]** The Rig runbook's fork section now describes two failure surfaces (BL-039's HTTP 400, and — post-BL-031 — a possible `stalled:` error surfacing through the same `Fork failed:` path). One sentence there pointing at the harness runbook's new section would close the loop for the Rig operator; the harness entry per §6 is landed and sufficient for the rule, so this is completeness, not compliance. Take or defer with a stated reason.
